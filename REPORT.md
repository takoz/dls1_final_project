Темой для финального проекта была выбрана задача детекции (продуктовый трек), а именно определение переломов на рентгеновских снимках.

# Выбор инструментов

В качестве моделей и фреймворка для обучения было решено взять `YOLO` и `ultralytics`. Быстрый поиск показал, что подобные задачи детекции довольно успешно решаются моделями `YOLOv8` с бекбоуном `ResNet50`. В дальнейшем для проекта были выбраны более современные версии `YOLO`, в частности `yolo26`. Кроме того, переоценить удобство использования `ultralytics` для дообучения модели довольно сложно. Это актуальный, хорошо поддерживаемый фреймворк с массой возможностей по кастомизации модели, настройке гиперпараметров и аугментаций, а также включающий множество удобных утилит для работы с датасетами и инференса.

В качестве фронта и бека для приложения был выбран `streamlit`. С первого взгляда стало понятно, что это крайне удобный инструмент для прототипирования приложений. Немаловажны также возможности для быстрого деплоя, которые дает `streamlit.io`.

# Датасет

Для исследования датасета был написан небольшой [скрипт на python](inspect_dataset.py). В первую очередь важно было посмотреть есть ли в датасете критическая для обучения несбалансированность классов. Кроме того скрипт дает возможность посмореть на любое изображение в датасете с наложенными bbox'ами.

Анализ показал, что датасет содержит всего несколько изображений для класса `humerus fracture`. Кроме того, более внимательное рассмотрение показало, что классы `humerus fracture` и `humerus` перепутаны. Было решено взять оригинальную версию этого датасета с [`roboflow`](https://universe.roboflow.com/bone-frac-dataset/bone-frac-dataset) - возможно какие-то изображения были потеряны при конвертации или есть более новая версия с большим количеством изображений. Датасет был сконвертирован в yolo-формат при помощи `ultralitics`:

```
from ultralytics.data.converter import convert_coco

convert_coco(
    labels_dir="BoneFractureCoco",
    save_dir="BoneFractureYoloOBB",
    use_segments=True
)
```

В новой версии стало заметно больше изображений для всех классов, но `humerus` по прежнему был представлен плохо. В итоге было решено использовать более свежую версию датасета, но исключить из нее класс `humerus`. Во-первых, нескольких изображений не достаточно для сколь-нибудь эффективного обучения, а, во-вторых, данная задача связана прежде всего с определением повреждений на снимках, а класс `humerus` -- это детекция плечевой кости самой по себе.

# Обучение

Обучение выполнялось на `kaggle`. В качестве моделей были выбраны `yolo26s-obb` (100 эпох), `yolo26n-obb` (50 эпох), `yolov8n-obb` (50 эпох). Все модели версий `OBB`, так как исходный датасет содержит полигоны вычисленные на основе сегментационных масок и оринтированный `bbox` позволит лучше максимизировать `IoU`.

В расчет были приняты следующие соображения:

- в список дефолтных аугментаций входит Blur, что не слишком полезно в анализе рентгеновских снимков, где нужно анализировать мелкие делали. Лучше избавиться от него
- стоит повысить вес `CLAHE`, так как это хорошо выделяет как раз мелкие детали, нужные модели
- стоит отключить mosaic, так как составление коллажей не поможет модели выучить особенности того или иного вида перелома
- остальные аугментации были немного подрегулированы исходя из общих соображений

**Результаты обучения**

| model         | epochs | mAP50 | mAP50-95 |
| ------------- | ------ | ----- | -------- |
| `yolo26s-obb` | 100    | 0.946 | 0.773    |
| `yolo26n-obb` | 50     | 0.885 | 0.584    |
| `yolov8n-obb` | 50     | 0.814 | 0.522    |

Хотя в целом можно было бы увеличить количество эпох до 150-200, возможно на одной из поздних стадий случился бы качественный скачок. Также, возможно большее количество эпох улучшило бы метрику `mAP50-95`.

# Инференс

Приложение доступно по адресу: [https://bonedetectionyolo.streamlit.app](https://bonedetectionyolo.streamlit.app)

В приложении доступны:

- выбор предобученной модели
- загрузка одного или нескольких изображений и детекция переломов на них. Для каждого результата сообщается скорость работы модели
- указание `Confidence Threshold` и `IoU Threshold`
- демо режим, когда выбранная модель работает с заранее загруженными изображениями - по одному для каждого класса. В этом режиме будет выведен не только результат детекции, но и `ground truth` для данного изображения взятый из датасета.

# Выводы

- модели обучились неплохо
- лучшие результаты ожидаемо показывает модель с большим количеством параметров обученная в течение большего количества эпох
- модели с меньшим количеством параметров ожидаемо быстрее, при этом качество детекции приемлемое
- слабым местом является исключение класса из датасета, лучше было найти больше изображений данного плечевой кости и добавить их в обучающую выборку
